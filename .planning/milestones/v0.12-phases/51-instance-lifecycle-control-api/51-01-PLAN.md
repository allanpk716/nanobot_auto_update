4. Method `HandleStart(w http.ResponseWriter, r *http.Request)`:
   - Extract `name` via `r.PathValue("name")`. If empty, return 400 `writeJSONError(w, http.StatusBadRequest, "bad_request", "Instance name required")`.
   - **Update-lock guard (addresses review HIGH-1):** Call `h.im.TryLockUpdate()`. If it returns false, return 409 `writeJSONError(w, http.StatusConflict, "conflict", "An update is already in progress, cannot start instance")`. If true, add `defer h.im.UnlockUpdate()` immediately.
   - Call `h.im.GetLifecycle(name)`. If error, return 404 `writeJSONError(w, http.StatusNotFound, "not_found", fmt.Sprintf("Instance %q not found", name))`.
   - Check `inst.IsRunning()`. If true, return 409 `writeJSONError(w, http.StatusConflict, "conflict", fmt.Sprintf("Instance %q is already running", name))`.
   - Create detached context: `ctx, cancel := context.WithTimeout(context.Background(), 60*time.Second)` with `defer cancel()`. The 60s outer timeout exceeds the default 30s startup_timeout from config, giving the inner logic room to control its own deadline. Document this as a comment: `// Outer timeout > startup_timeout to let inner logic control its own deadline`.
   - Call `inst.StartAfterUpdate(ctx)`. If error, log with `h.logger.Error("Failed to start instance", "instance", name, "error", err)` and return 500 `writeJSONError(w, http.StatusInternalServerError, "internal_error", fmt.Sprintf("Failed to start instance %q: %v", name, err))`.
   - On success, log with `h.logger.Info("Instance started via API", "instance", name)` and return 200 JSON with exact schema: `{"message": "Instance '<name>' started", "running": true}`. Use `w.Header().Set("Content-Type", "application/json")` before `json.NewEncoder(w).Encode(...)`.

5. Method `HandleStop(w http.ResponseWriter, r *http.Request)`:
   - Extract `name` via `r.PathValue("name")`. If empty, return 400 `writeJSONError(w, http.StatusBadRequest, "bad_request", "Instance name required")`.
   - **Update-lock guard (addresses review HIGH-1):** Call `h.im.TryLockUpdate()`. If it returns false, return 409 `writeJSONError(w, http.StatusConflict, "conflict", "An update is already in progress, cannot stop instance")`. If true, add `defer h.im.UnlockUpdate()` immediately.
   - Call `h.im.GetLifecycle(name)`. If error, return 404 `writeJSONError(w, http.StatusNotFound, "not_found", fmt.Sprintf("Instance %q not found", name))`.
   - Check `inst.IsRunning()`. If false, return 409 `writeJSONError(w, http.StatusConflict, "conflict", fmt.Sprintf("Instance %q is not running", name))`.
   - Create context: `ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)` with `defer cancel()`. The 30s is > the 5s stop timeout used internally by `StopForUpdate`, giving the inner logic room to complete. Document as comment: `// Outer timeout > inner stopTimeout (5s) to let inner logic control its own deadline`.
   - Call `inst.StopForUpdate(ctx)`. If error, log with `h.logger.Error("Failed to stop instance", "instance", name, "error", err)` and return 500 `writeJSONError(w, http.StatusInternalServerError, "internal_error", fmt.Sprintf("Failed to stop instance %q: %v", name, err))`.
   - On success, log with `h.logger.Info("Instance stopped via API", "instance", name)` and return 200 JSON with exact schema: `{"message": "Instance '<name>' stopped", "running": false}`.

**Success response JSON schema (addresses review HIGH-2):**
- Start 200: `{"message": "Instance 'x' started", "running": true}`
- Stop 200: `{"message": "Instance 'x' stopped", "running": false}`
- Both use `fmt.Sprintf("Instance %q started/stopped", name)` which produces quoted name in the message string.
- Note: The restart handler in `internal/web/handler.go` uses `{"success": true}`. This lifecycle handler uses a different format (`message` + `running`) because the UI needs to know the running state after the operation. The `running` field is critical for the Phase 53 UI to update status indicators without an additional GET request.

Key design decisions (from RESEARCH + review feedback):
- **Update-lock coordination (review HIGH-1):** `TryLockUpdate()`/`defer UnlockUpdate()` prevents races with `TriggerUpdate` and `SelfUpdateHandler`. If an update is in progress (isUpdating=true), the lock acquisition fails and we return 409 immediately. This mirrors how `TriggerUpdate` and `SelfUpdateHandler` coordinate via the same `atomic.Bool`.
- Use `context.Background()` (not `r.Context()`) for start operations to prevent orphaned processes on client disconnect (RESEARCH Pitfall 3).
- Return 409 Conflict for wrong-state operations, not 200 (per RESEARCH Open Question 1 recommendation).
- Always use `writeJSONError()` for error responses, never `http.Error()` (per RESEARCH Anti-Pattern).
- Do NOT call lower-level lifecycle functions directly; always use `StartAfterUpdate()` / `StopForUpdate()` which handle Telegram monitor lifecycle (Pitfall 5).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> API | Untrusted HTTP input crosses into authenticated lifecycle handlers |
| API -> InstanceManager | Handler calls GetLifecycle and TryLockUpdate which access internal instance state |
| API -> Process | StartAfterUpdate/StopForUpdate manage OS-level processes |
| Lifecycle ops -> Update lock | TryLockUpdate/UnlockUpdate coordinates with TriggerUpdate and SelfUpdate |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-51-01 | Spoofing | Lifecycle endpoints | mitigate | AuthMiddleware with constant-time Bearer token comparison (reuses existing mechanism) |
| T-51-02 | Tampering | Start command injection | accept | Start command comes from config (not user input); handler never accepts command from HTTP body |
| T-51-03 | Denial of Service | Concurrent start of same instance | mitigate | (1) IsRunning() check prevents double-start; (2) TryLockUpdate() prevents concurrent operations during updates |
| T-51-04 | Denial of Service | Start/stop racing with TriggerUpdate/SelfUpdate | mitigate | TryLockUpdate()/UnlockUpdate() serializes lifecycle ops with bulk updates via shared atomic.Bool |
| T-51-05 | Information Disclosure | Timing attack on auth token | mitigate | subtle.ConstantTimeCompare in existing AuthMiddleware (no new code needed) |
</threat_model>
