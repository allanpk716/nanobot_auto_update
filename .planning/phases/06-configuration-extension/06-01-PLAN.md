  <action>
扩展 `internal/config/config.go` 添加多实例支持:

1. **扩展 Config 结构体**:
   ```go
   type Config struct {
       Cron      string           `mapstructure:"cron"`
       Nanobot   NanobotConfig    `mapstructure:"nanobot"`    // 保留向后兼容
       Instances []InstanceConfig `mapstructure:"instances"`  // 新增
       Pushover  PushoverConfig   `mapstructure:"pushover"`
   }
   ```

2. **实现 ValidateModeCompatibility() 方法**:
   - 检测 Nanobot.Port != 0 (表示用户配置了旧模式)
   - 检测 len(c.Instances) > 0 (表示用户配置了新模式)
   - 如果两者同时存在,返回错误: `配置错误: 不能同时使用 'nanobot' section 和 'instances' 数组,请选择其中一种配置模式`

3. **实现 validateUniqueNames() 函数**:
   - 使用 map[string]int 记录名称到索引的映射
   - 检测重复名称时返回详细错误: `配置验证失败: 实例名称重复 - "name1" 出现在第 2 和第 5 个实例配置中`
   - 索引从 1 开始计数 (面向用户)

4. **实现 validateUniquePorts() 函数**:
   - 使用 map[uint32]string 记录端口到实例名称的映射
   - 检测重复端口时返回详细错误: `配置验证失败: 端口重复 - 18790 出现在实例 "instance1" 和 "instance2" 中`

5. **重构 Config.Validate() 方法**:
   - 使用 `var errs []error` 收集所有错误
   - 先验证 cron 表达式
   - 调用 ValidateModeCompatibility()
   - 根据模式选择验证路径:
     - 如果 len(c.Instances) > 0: 调用 validateUniqueNames, validateUniquePorts, 和每个实例的 Validate()
     - 否则: 调用 c.Nanobot.Validate()
   - 使用 `errors.Join(errs...)` 聚合所有错误

6. **更新 Load() 函数**:
   - 保持现有 defaults 设置不变
   - viper.Unmarshal 自动处理 instances 数组 (无需额外代码)
   - Validate() 调用会处理新的验证逻辑

**避免的做法**:
- 不要在 Validate() 中遇到第一个错误就返回 (使用 errors.Join 聚合)
- 不要使用 O(n²) 嵌套循环检查唯一性 (使用 map)
- 不要破坏现有单实例配置的验证路径
