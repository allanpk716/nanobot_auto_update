Add an `UpdateConfig` function to `internal/config/config.go` that atomically performs the full read-modify-write cycle. This solves both the race condition (two concurrent creates overwriting each other) and the shared-state mutation problem (handler corrupts `globalHotReload.current` on save failure).

**Addresses review concerns:**
- HIGH (both reviewers): Read-modify-write race condition -- `configMu` only protected viper write, not the full cycle
- HIGH (both reviewers): Shared state mutation without deep copy -- `GetCurrentConfig()` returns shared pointer, handler mutates it directly

Implementation:

1. Add a `sync.Mutex` package-level var `updateMu` to serialize the entire read-modify-write cycle. Name it `updateMu` (not `configMu`) to distinguish from the hot-reload mutex.

2. Add a `deepCopyConfig` helper function that creates a value-level copy of `*Config`, including a new slice for `Instances` so that `append` does not modify the original backing array:

```go
// updateMu serializes the full read-modify-write cycle for config updates.
// This prevents concurrent API requests from causing data loss.
var updateMu sync.Mutex

// deepCopyConfig creates a deep copy of the Config struct.
// The Instances slice is recreated so mutations to it do not affect the original.
func deepCopyConfig(cfg *Config) *Config {
    if cfg == nil {
        return nil
    }
    copy := *cfg
    copy.Instances = make([]InstanceConfig, len(cfg.Instances))
    for i := range cfg.Instances {
        copy.Instances[i] = cfg.Instances[i]
        // Deep copy AutoStart pointer
        if cfg.Instances[i].AutoStart != nil {
            val := *cfg.Instances[i].AutoStart
            copy.Instances[i].AutoStart = &val
        }
    }
    return &copy
}

// UpdateConfig atomically reads the current config, applies a mutation function, and persists the result.
// The mutation function receives a deep copy of the current config -- it is safe to modify freely.
// If the mutation function returns nil, the modified copy is written to disk via viper.
// If the mutation function returns an error, no write occurs and the original config is untouched.
// If the viper write fails, no write occurs and the original config is untouched.
//
// This function serializes all callers via updateMu, so concurrent API requests are safe.
//
// D-08: Uses viper.WriteConfig() to persist to the same file viper loaded from.
// D-09: After writing, hot-reload watcher detects the file change (500ms debounce).
func UpdateConfig(fn func(*Config) error) error {
    updateMu.Lock()
    defer updateMu.Unlock()

    // Read current config (this is a pointer to globalHotReload.current)
    current := GetCurrentConfig()
    if current == nil {
        return fmt.Errorf("config not initialized (WatchConfig not started)")
    }

    // Deep copy so mutations don't affect the live config
    copy := deepCopyConfig(current)

    // Apply the caller's mutation
    if err := fn(copy); err != nil {
        return fmt.Errorf("mutation failed: %w", err)
    }

    // Persist the modified copy
    v := GetViper()
    if v == nil {
        return fmt.Errorf("viper not initialized")
    }
    v.Set("instances", copy.Instances)
    if err := v.WriteConfig(); err != nil {
        return fmt.Errorf("failed to write config file: %w", err)
    }

    return nil
}
```

3. Add `"sync"` to imports if not already present (it is already imported in hotreload.go but check config.go).

**Key design points:**
- `updateMu` is separate from `globalHotReload.mu` -- the hot-reload lock protects reads of `globalHotReload.current`, `updateMu` protects the full cycle. Nested locking order: `updateMu` first, then `GetCurrentConfig` acquires `globalHotReload.mu` internally. This is safe because `doReload` only acquires `globalHotReload.mu` (never `updateMu`), so no deadlock.
- `deepCopyConfig` creates a new `Instances` slice so `append` does not corrupt the original backing array. Each `AutoStart *bool` is individually copied.
- The handler no longer calls `GetCurrentConfig()` + `SaveConfig()` separately. It calls `UpdateConfig(func(cfg *Config) error { ... })` which handles everything atomically.
- If the mutation function returns an error (e.g., validation failure), nothing is written.
- If `viper.WriteConfig()` fails, the in-memory `globalHotReload.current` is untouched because we never mutated it.
